## Feature 1: Semantic Chunking for Long Documents

**Problem:** Current chunkers (TextAdapter, MarkdownAdapter) split by paragraphs or headings. A 50-page PDF gets split into 200+ tiny memories with no semantic coherence. Recall finds fragments, not concepts.

**Solution:** Add an optional semantic chunking pass that groups related paragraphs before creating memories. Use SBERT embeddings to cluster paragraphs by cosine similarity, then merge clusters into coherent chunks of ~500-1000 words.

**Files:**
- `src/cerebro/ingestion/chunker.py` — add `SemanticChunker` class
- `src/cerebro/ingestion/text_adapter.py` — integrate semantic chunking
- `src/cerebro/ingestion/markdown_adapter.py` — integrate semantic chunking
- `src/cerebro/config.py` — add `SEMANTIC_CHUNKING_ENABLED`, `SEMANTIC_CHUNK_SIZE`, `SEMANTIC_CHUNK_OVERLAP`
- `src/cerebro/ingestion/pipeline.py` — pass chunking config to adapters

**Work:**
- [ ] `SemanticChunker` class: embed paragraphs, cluster by similarity threshold, merge clusters
- [ ] Configurable chunk size (target tokens/words) and overlap (sliding window)
- [ ] Opt-in via config (`SEMANTIC_CHUNKING_ENABLED = False` by default)
- [ ] Falls back to paragraph splitting if SBERT fails or doc is short
- [ ] Tests in `tests/test_ingestion/test_chunker.py`

**Design note:** Use existing SBERT model (already loaded in `embeddings.py`). No new dependencies.

## Feature 2: Near-Duplicate Detection at Ingestion Time

**Problem:** `NEAR_DEDUP_THRESHOLD = 0.95` exists in config.py but is NOT used during ingestion. Dropping the same file twice creates duplicate memories. The threshold is referenced nowhere in the ingestion pipeline.

**Solution:** Before creating a new memory during ingestion, check ChromaDB for existing memories with cosine similarity > threshold to the new embedding. If found, skip creation and optionally strengthen the existing memory's links.

**Files:**
- `src/cerebro/cortex.py` — add `find_near_duplicates(embedding, threshold)` helper
- `src/cerebro/ingestion/pipeline.py` — call dedup check before `cortex.remember()`
- `src/cerebro/storage/chroma_store.py` — add `query_by_embedding()` for raw vector search
- `src/cerebro/config.py` — `NEAR_DEDUP_THRESHOLD` already exists, ensure default is sensible
- `src/cerebro/interfaces/api_server.py` — add `?dedup=false` override to `/ingest/upload`

**Work:**
- [ ] `CerebroCortex.find_near_duplicates(embedding, threshold)` → returns list of matching memory IDs
- [ ] `IngestionPipeline.ingest_file()` → checks dedup before each `remember()` call
- [ ] On duplicate found: skip creation, log to event bus, optionally add `supports` link
- [ ] Configurable: `DEDUP_AT_INGESTION = True` by default
- [ ] Dashboard: show "X duplicates skipped" in ingest panel progress
- [ ] Tests in `tests/test_ingestion/test_adapters.py`

**Design note:** Use ChromaDB's `query()` with the raw embedding — no new infrastructure. Fast (<10ms per check).

## Feature 3: Temporal Decay Visualization in Dashboard

**Problem:** The activation system (ACT-R + FSRS) is invisible. Users can't see which memories are fading, which are strengthening, or how the decay curves look. The dashboard shows counts but no dynamics.

**Solution:** Add a "Decay" panel to the dashboard showing:
- Scatter plot: memories by age vs. activation level (colored by layer)
- Decay curve: theoretical ACT-R curve for a memory with N accesses
- "At risk" list: memories in sensory/working layer that haven't been accessed in >24h
- Layer transition timeline: when memories were promoted/demoted

**Files:**
- `src/cerebro/web/js/panels/decay.js` — new panel
- `src/cerebro/web/index.html` — add Decay nav button
- `src/cerebro/interfaces/api_server.py` — add `/activation/heatmap` and `/activation/at-risk` endpoints
- `src/cerebro/activation/strength.py` — add `compute_activation_curve(node, days=30)` helper
- `src/cerebro/cortex.py` — add `get_at_risk_memories(hours=24, layer=None)` method

**Work:**
- [ ] Backend: `/activation/heatmap` → returns {memory_id, age_hours, activation, layer, salience}[]
- [ ] Backend: `/activation/at-risk` → returns memories with low activation + old age
- [ ] Frontend: Scatter plot using Chart.js or raw canvas (keep it lightweight)
- [ ] Frontend: Decay curve visualization — B(t) = ln(Σ t_k^{-d}) plotted over time
- [ ] Frontend: "At risk" table with revive button (triggers a `recall()` to boost activation)
- [ ] Tests in `tests/test_interfaces/test_api_server.py`

**Design note:** Use vanilla canvas for plotting — no new JS dependencies. The particle canvas in app.js proves this approach works.

## Feature 4: Audit Logging for Access Denials and Visibility Changes

**Problem:** Multi-agent setups have no audit trail. If BOB tries to access ALICE's private memory, it fails silently. If someone changes visibility from SHARED to PRIVATE, there's no record. This is a security/ops gap.

**Solution:** Add an `audit_log` table to SQLite. Log:
- Access denials (who tried to access what, why it was denied)
- Visibility changes (who changed what, old→new visibility)
- Memory edits (who edited what, content diff summary)
- Link creation/deletion (who linked what)

**Files:**
- `src/cerebro/storage/sqlite_schema.py` — add `audit_log` table (schema v7 migration)
- `src/cerebro/cortex.py` — add `log_audit(event_type, actor, target, details)` method
- `src/cerebro/storage/graph_store.py` — add `append_audit()` method
- `src/cerebro/interfaces/api_server.py` — add `/audit/log` and `/audit/query` endpoints
- `src/cerebro/web/js/panels/audit.js` — new panel
- `src/cerebro/web/index.html` — add Audit nav button

**Work:**
- [ ] Schema: `audit_log` table with id, timestamp, event_type, actor_agent_id, target_memory_id, old_value, new_value, details_json
- [ ] Hook into `_can_access()` — on denial, log to audit_log
- [ ] Hook into `share_memory()` — log visibility change
- [ ] Hook into `update_memory()` — log content edit (store old content hash, not full text)
- [ ] Hook into `associate()` and `delete_memory()` — log link and deletion events
- [ ] API: `GET /audit/query?event_type=denial&actor=BOB&limit=50`
- [ ] Dashboard: Audit panel with filterable table
- [ ] Config: `AUDIT_LOGGING_ENABLED = True` by default
- [ ] Tests in `tests/test_storage/test_audit.py`

**Design note:** Keep audit entries small — store content hashes, not full text. Use JSON column for flexible details. Purge old audits via `cerebro audit --purge-days 90` CLI command.
